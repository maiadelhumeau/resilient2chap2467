Resilient Web Design

Chapter 2: Materials

Au risque d'apprendre aux vieux singes à faire la grimace, j'aimerais que vous réfléchissiez à ce qui se passe lorsqu'un navigateur analyse un élémentHTML. Prenez, par exemple, un élément de paragraphe contenant du texte. Il y a une balise P d'ouverture, une balise P de fermeture, et entre ces balises, il y a le texte. .

<p>du texte</p>

Un navigateur Web rencontrant cet élément affichera le texte entre les balises d'ouverture et de fermeture. Considérez maintenant ce qui se passe lorsque ce même navigateur Web rencontre un élément qu'il ne reconnaît pas.

<marklar>encore du texte</marklar>

Une fois de plus, le navigateur affiche le texte entre les balises d'ouverture et de fermeture. Ce qui est intéressant ici, c'est ce que le navigateur ne fait pas. Le navigateur ne renvoie pas d'erreur. Le navigateur n'arrête pas d'analyser le HTML à ce stade, refusant d'aller plus loin. Au lieu de cela, il ignore simplement les balises et affiche le contenu à l'intérieur.

Cette attitude libérale vis-à-vis des erreurs a permis au vocabulaire duHTML de croître avec le temps, passant des 21 éléments d'origine aux 121 éléments du HTML5. Chaque fois qu'un nouvel élément est introduit dans HTML, we know exactly how older browsers will treat it; they will ignore the tags and display the content.

C'est une fonctionnalité remarquablement puissante. Il permet aux navigateurs d'implémenter de nouvelles fonctionnalitésHTML à des rythmes différents. Nous n'avons pas besoin d'attendre que chaque navigateur reconnaisse un nouvel élément. Au lieu de cela, nous pouvons commencer à utiliser le nouvel élément à tout moment, en sachant que les navigateurs non compatibles ne s'y étoufferont pas.

<main>ce texte s’affichera dans n’importe quel navigateur</main>

Si les navigateurs Web traitent toutes les balises de la même manière - en affichant leur contenu - alors à quoi sert d'avoir un vocabulaire d'éléments en HTML?

La signification du balisage

Certains éléments HTML n'ont littéralement aucun sens. L'élémentSPAN ne dit rien sur le contenu qu'il contient. En ce qui concerne un navigateur Web, vous pouvez également utiliser un élément MARKLAR inexistant. Mais c'est l'exception. La plupart des élémentsHTML existent pour une raison. Ils ont été créés et approuvés afin de rendre compte de situations spécifiques que des auteurs comme vous et moi sommes susceptibles de rencontrer.

Il y a évidemment des éléments spéciaux, comme l'élément A, qui sont fournis avec des super pouvoirs. Dans le cas de l'élément A, sa superpuissance réside dans l'attributHREF qui nous permet de nous connecter à n'importe quelle autre ressource sur le Web. D'autres éléments comme INPUT, SELECT, TEXTAREA, et BUTTON ont leurs propres super-pouvoirs, ce qui permet aux gens d'entrer des données et de les soumettre à un serveur Web.

Ensuite, il y a des éléments qui décrivent le type de contenu qu'ils contiennent. Le contenu d'un élément P doit être considéré comme un paragraphe de texte. Le contenu d'un élément LI doit être considéré comme un élément d'une liste. Les navigateurs affichent le contenu de ces éléments avec quelques indices visuels quant à leur signification. Les paragraphes sont affichés avec des espaces avant et après leur contenu. Les éléments de la liste sont affichés avec des puces ou des numéros avant leur contenu.

Le début de la croissance du vocabulaire HTML a été rempli de nouveaux éléments qui ont fourni des instructions visuelles aux navigateurs Web: BIG, SMALL, CENTER, FONT. En fait, les instructions visuelles étaient la seule raison pour laquelle ces éléments existent ne fournissent aucune indication sur la signification du contenu qu'elles contenaient. Le HTML risquait de devenir un langage d'instruction visuelle au lieu d'un vocabulaire de sens.

Une question de style

Håkon Wium Lie travaillait au CERN en même temps que Tim Berners-Lee. Il a immédiatement reconnu le potentiel du World Wide Web et de son langage, HTML. Il s'est également rendu compte que le pouvoir expressif de la langue risquait d'être submergé par des traits visuels. Lee a proposé un nouveau format pour décrire la présentation des documents HTML documents: les feuilles de style en cascade.

Il est rapidement rejoint par le programmeur néerlandais Bert Bos. Ensemble, ils se sont mis à créer une syntaxe suffisamment puissante pour gérer les demandes des concepteurs, tout en restant suffisamment simple pour apprendre rapidement. Ils ont réussi.

Pensez un instant à tous les sites disponibles sur le Web. Il existe une énorme variation dans le style visuel: jeux de couleurs, traitements typographiques, textures et mises en page. Toute cette variété est rendue possible par un modèle simple qui décrit tout le CSSjamais écrit:

selector {
    property: value;
}

C'est tout.

CSS a partagé l'attitude indulgente du HTML face aux erreurs. Si un navigateur Web rencontre un sélecteur qu'il ne comprend pas, il ignore simplement tout ce qui se trouve entre les accolades de ce sélecteur. Si un navigateur voit une propriété ou une valeur qu'il ne comprend pas, il ignore simplement cette déclaration particulière. Le navigateur ne renvoie pas d'erreur. Le navigateur n'arrête pas d'analyser le CSS à ce stade, refusant d'aller plus loin.

marklar {
    marklar: marklar;
}

Tout comme avec HTML, cette gestion des erreurs lâche a permis au CSS de se développer au fil du temps. De nouveaux sélecteurs, de nouvelles propriétés et de nouvelles valeurs ont été ajoutés au vocabulaire de la langue au fil des ans. Chaque fois qu'une nouvelle fonctionnalité arrive dans CSS, les concepteurs et les développeurs savent qu'ils peuvent l'utiliser en toute sécurité, même si elle n'est pas encore largement prise en charge dans les navigateurs. Ils peuvent être assurés que les anciens navigateurs réagiront aux nouvelles fonctionnalités avec une indifférence totale.

Ce n'est pas parce qu'une langue est élégante et bien conçue que les gens l'utiliseront. Le CSS est arrivé plus tard que le HTML. Les concepteurs n'ont pas passé les années qui ont suivi à attendre patiemment un moyen de styliser leurs documents sur le Web. Ils ont utilisé ce qui était à leur disposition.

Déchire tout

En 1996 David Siegel a publié un livre intitulé Creating Killer Websites. A l'intérieur, il y décrit une série de techniques ingénieuses pour éliminer des conceptions accrocheuses à partir de la matière première du HTML.

Une technique impliquait l'utilisation d'un GIF transparent, juste un pixel par un pixel. Si cela était inséré dans une page en tant qu'élément IMG mais avec des valeurs précises dans ses attributs WIDTH et HEIGHT, les concepteurs pouvaient contrôler la quantité d'espaces blancs dans leurs conceptions.

Une autre technique a utilisé l'élément TABLE. Cet élément - avec ses enfants TR and TD— était destiné à décrire des données tabulaires. Mais avec les bonnes valeurs appliquées aux largeurs et hauteurs des cellules de tableau, il pourrait être utilisé pour recréer à peu près n'importe quelle disposition souhaitée.

C'étaient des hacks; des solutions intelligentes à des problèmes délicats. Mais ils ont eu des conséquences malheureuses. Les concepteurs traitaient le HTML comme un outil pour l'apparence du contenu au lieu d'un langage pour décrire la signification du contenu. CSS était une solution à ce problème, si seulement les concepteurs pouvaient être convaincus de l'utiliser.

GUERRES DES NAVIGATEURS

One of the reasons why web designers weren’t using CSS was the lack of browser support. Back then there were two major browsers competing for the soul of the web: Microsoft Internet Explorer and Netscape Navigator. They were incompatible by design. One browser would invent a new HTML element or attribute. The other browser would invent their own separate element or attribute to do exactly the same thing.

Perhaps the thinking behind this strategy was that web designers would have to choose which proprietary features they were going to get behind, like children being forced to choose between parents. In reality web designers had little choice but to write for both browsers which meant doing twice the work.

A group of web designers decided enough was enough. They gathered together under the banner of the Web Standards Project and began lobbying Microsoft and Netscape to abandon their proprietary ways and adopt standards such as CSS.

The tide began to turn with the launch of Internet Explorer 5 for the Mac, a browser that shipped with impressive CSS support. If this was the future of web design, life was about to get a lot more productive and creative.

Some forward‐thinking web designers caught the CSS bug early. They redesigned their websites using CSS for layout instead of using TABLEs and spacer GIFs. True to the founding spirit of the web, they shared what they were learning and encouraged others to make the switch to CSS.

Perhaps the best demonstration of the power of CSS was a website called the CSS Zen Garden, created by Dave Shea. It was a showcase of beautiful and varied designs, all of them accomplished with CSS. Crucially, the HTML remained the same.

Seeing the same HTML document styled in a multitude of different ways drove home one of the beneficial effects of CSS: separation of concerns.

Coupling

In any system, from urban infrastructure to a computer program, the designers of that system can choose the degree to which the pieces of the system depend on one another. In a tightly‐coupled system, every piece depends on every piece. In a loosely‐coupled system, all the pieces are independent, with little to no knowledge of the other pieces.

In a tightly‐coupled system, each part of the system can make assumptions about the other parts. These systems can be designed quite quickly, but at a price. They lack resilience. If one piece fails, it could take the whole system with it.

Designing a loosely‐coupled system can take more work. The payoff is that the overall result is more resilient to failure. Individual parts of the system can be swapped out with a minimum of knock‐on effects.

The hacks pioneered by David Siegel tightly coupled structure and presentation into a single monolithic HTML file. The adoption of CSS eased this dependency, bringing the web closer to the modular approach of the UNIX philosophy. The presentational information could be moved into a separate file: a style sheet. That’s how a single HTML document at the CSS Zen Garden could have so many different designs applied to it.

The style sheet still needs to have some knowledge of the HTML document’s structure. Quite often, “hooks” are added into the markup to make it easier to style: specific values of CLASS or ID attributes, for example. So HTML and CSS aren’t completely decoupled. They form a partnership but they also have an arrangement. The markup document might decide that it wants to try seeing other style sheets sometimes. Meanwhile, the style sheet could potentially be applied to other documents. They are loosely coupled.

Dancing about architecture

It takes time for a discipline to develop its own design values. Web design is a young discipline indeed. While we slowly begin to form our own set of guiding principles, we can look to other disciplines for inspiration.

The world of architecture has accrued its own set of design values over the years. One of those values is the principle of material honesty. One material should not be used as a substitute for another. Otherwise the end result is deceptive.

Using TABLEs for layout is materially dishonest. The TABLE element is intended for marking up the structure of tabular data. The end result of using TABLEs, FONT elements, and spacer GIFs is a façade. At first glance everything looks fine, but it won’t stand up to scrutiny. As soon as such a website is stress‐tested by actual usage across a range of browsers, the façade crumbles.

Using CSS for presentation is materially honest—that’s the intended use of CSS. It also allows HTML to be materially honest. Instead of trying to fulfil two roles—structure and presentation—HTML can return to fulfilling its true purpose, marking up the meaning of content.

It’s still possible to use (or abuse) CSS to be materially dishonest. For the longest time, there was no easy way to add rounded corners to an element using CSS. Instead, web designers found ways to hack around the problem, putting background images on the element to simulate the same end effect. It worked up to a point, but just like the spacer GIF hack, it was a façade. Then the border‐radius property arrived. Now designers can have their rounded corners in a materially honest way.

Crucially, designers were able to use new properties like border‐radius long before every web browser supported them. That’s all thanks to the liberal error‐handling model of CSS. Newer browsers would display the rounded corners. Older browsers would not throw an error. Older browsers would not stop parsing the CSS and refuse to parse any further. They would simply ignore the instructions they didn’t understand and move on. No harm, no foul.

Of course this means that the resulting website will look different in different browsers. Some people will see rounded corners. Others won’t.

And that’s okay.
